# Ask Intent Router 与可编辑预览设计

## 目标

在现有 `Ask` 基础上补齐三块关键能力：

1. 把基于关键词的动作触发升级为更通用的 `Intent Router`
2. 把“发给谁 / 发到哪个群”的解析升级为可选目标推荐，而不是直接 fallback
3. 把中高风险动作统一收成“可编辑预览卡 -> 确认 / 取消 -> 执行”

这次设计要同时满足两个要求：

- 保持高扩展性和模块化，不能继续把逻辑写死在 `HR Recruiting Skill`
- 不破坏现有 `/knowledge` 与 `/ask` 的分工，以及现有飞书动作执行链

## 背景问题

当前 `/ask` 已经具备：

- 知识问答的 `search -> rerank -> answer -> citations`
- HR recruiting 场景下的简历、面试、反馈、协同动作
- 飞书绑定、预览、确认、执行基础链路

但仍有三个明显问题：

1. 动作触发仍偏关键词驱动  
   例如提到“面试”就容易进入推荐时段路径，缺少对用户完整意图的理解。

2. 目标解析仍偏硬编码  
   当消息目标没有解析清楚时，会 fallback 到默认群，导致用户感觉“写死了默认群”。

3. 预览卡不够完整  
   当前虽然存在确认卡，但更像“确认执行”入口，而不是“可编辑的动作预览”。

## 设计原则

1. `Intent Router` 独立  
   意图识别不再由 HR skill 内部关键词直接决定，而是由独立路由层输出标准化意图。

2. `Target Resolver` 独立  
   联系人 / 群 / 会话目标的检索与推荐不写死在 Skill 内部。

3. `Preview First`  
   所有 `medium / high` 风险动作，都必须先生成预览卡，允许用户编辑，再进入执行。

4. `Skill / Tool 解耦`
   - Skill 负责理解任务、生成动作草案
   - Tool 负责访问飞书和执行动作
   - Ask runtime 负责路由和组合输出

5. `显式目标优先`
   目标解析优先级固定为：
   - 用户显式指定目标
   - 会话上下文中的已确认目标
   - 推荐候选供用户选择
   - 绝不在中高风险动作里静默 fallback 到默认群

## 总体架构

在现有 Ask runtime 上新增三层：

1. `Intent Router`
- 输入：用户消息、最近上下文、当前 skill
- 输出：
  - `knowledge_qa`
  - `action_only`
  - `answer_then_action`
  - `clarification`

2. `Action Planner`
- 输入：路由结果、上下文、工具目录
- 输出结构化 `action_draft`
- 只负责：
  - 动作类型
  - 目标类型
  - 参数草案
  - 风险级别
  - 预览文案

3. `Target Resolver`
- 输入：目标查询文本，例如“李雷”“Ai应用开发群”
- 输出：
  - Top 3 推荐
  - 一个 `其他` 选项

现有 `Executor Runtime` 保持不变，仍然走：

`preview -> confirm -> execute`

## Intent Router

### 目标

把“看到关键词就进入某个分支”的逻辑收成更稳的意图识别层。

### 输入

- 当前消息
- 最近 3 到 5 轮对话摘要
- 当前 `active_context`
- 当前 skill

### 输出

#### 1. `knowledge_qa`

适用于：

- “报销额度是多少”
- “多模态是什么”
- “这个制度的审批流程是什么”

如果是企业问题，后续优先走 grounded RAG。  
如果是通用问题，后续允许走 general LLM。

#### 2. `action_only`

适用于：

- “发消息给李雷”
- “发到 Ai应用开发群”
- “帮我建个会”

这类不需要先生成答案，只需要进入动作草案和预览。

#### 3. `answer_then_action`

适用于：

- “把刚才的结论发给李雷”
- “先告诉我报销额度，再发到群里”
- “把这个候选人不错发送到 Ai应用开发群”

先生成可分享内容，再进入动作预览。

#### 4. `clarification`

适用于：

- 目标不明确
- 参数缺失
- 上下文不完整

例如：

- “发给他”
- “安排一下面试”
- “发到群里”

## Target Resolver

### 目标

用户说“发给谁 / 发到哪个群”时，系统要给候选，而不是直接执行或偷偷 fallback。

### 数据源

第一版优先从飞书工具获取：

- 联系人搜索
- 群聊搜索
- P2P 会话搜索

### 输出形式

当目标查询有歧义或尚未确认时，返回 `target_selection_card`：

- Top 3 推荐目标
- 一个 `其他` 选项

### Top 3 推荐规则

按以下顺序选推荐：

1. 精确名称匹配优先
2. 活跃上下文目标优先
3. 最近会话使用过的目标优先
4. 其余按工具返回顺序

### `其他` 选项

用户点击 `其他` 后，不强制弹独立页面。  
第一版采用聊天内补充方式：

- 卡片提示：`请在输入框里补充目标名称`
- 用户直接继续在聊天框输入目标

这样实现成本更低，也更符合现有 Ask 交互。

### 行为边界

对于 `medium / high` 风险动作：

- 如果目标没有确认，不允许直接执行
- 不允许静默 fallback 到默认群

对于 `low` 风险内部辅助动作：

- 仍允许默认群作为显式配置的默认建议值
- 但必须在预览卡里展示出来

## 可编辑预览卡

### 目标

预览卡不只是“确认一下”，而是让用户清楚看到将要发生什么，并允许必要编辑。

### 适用范围

所有 `medium / high` 风险动作都必须先进入可编辑预览卡：

- 给人发消息
- 发群消息
- 邀请他人参会
- 多动作组合执行

### 卡片内容

预览卡至少展示：

- 动作类型
- 目标对象
- 正文内容
- 时间信息
- 将调用的飞书工具
- 风险级别
- 可能的执行结果说明

### 可编辑字段

第一版允许编辑：

- 消息目标
- 消息正文
- 会议标题
- 会议开始 / 结束时间

### 用户操作

预览卡统一支持：

- `确认执行`
- `取消`
- `编辑后再确认`

### 编辑交互

第一版不做复杂弹窗编辑器。  
采用卡片内轻量字段编辑：

- 文本框编辑正文
- 下拉或按钮重新选目标
- 时间字段支持重新选择推荐时段

修改后的值写回 `pending_action_draft`，再点击确认时按最新值执行。

## Action Planner

### 目标

把现在散落在 HR skill 内部的动作拼接逻辑抽成通用草案层。

### 标准输出

`action_draft` 至少包含：

- `intent`
- `risk_level`
- `actions`
- `preview`
- `missing_fields`
- `resolved_targets`

### 示例

用户：

`发信息给李雷，告诉他面试通过了`

输出草案：

- intent: `message.send`
- target query: `李雷`
- candidate targets: Top 3 + 其他
- draft text: `你好，面试已通过，后续安排会继续同步给你。`
- risk level: `medium`
- requires_preview: `true`

### 多动作场景

例如：

`明天下午 3 点开 30 分钟会，并把刚才的提纲发到 Ai应用开发群`

Action Planner 产出：

1. `calendar.schedule`
2. `message.send`

第二个动作可以依赖第一个动作生成的会议链接。

## 上下文管理

### 分层上下文

Ask 会话上下文分成三层：

1. `conversation_memory`
- 最近几轮消息
- 用于意图理解和代词消解

2. `working_context`
- 当前候选人
- 当前岗位
- 当前知识问答结果
- 当前已确认目标

3. `pending_action_draft`
- 当前待确认动作草案
- 当前可编辑预览内容

### 写回规则

- 问答结束后，若答案可复用，写入 `last_shareable_text`
- 目标确认后，写入 `last_confirmed_target`
- 预览编辑后，更新 `pending_action_draft`

### 设计结论

这样可以避免：

- 旧默认群污染新动作
- 上一轮内容误被当作本轮正文
- 确认卡状态与实际执行参数不一致

## 与知识库的关系

### 页面分工

- `/knowledge`
  - 继续作为知识库管理台
  - 用于上传、管理、调试检索和内容质量

- `/ask`
  - 作为统一对话入口
  - 可直接使用同一套知识检索和 RAG 能力

### Ask 内部行为

当用户在 `/ask` 中直接提问：

- “报销额度是多少”

系统行为应为：

1. 路由到 `knowledge_qa`
2. 检索知识库
3. rerank
4. 交给生成模型回答
5. 返回答案和引用

也就是说：

知识库管理和知识库使用分离，但底层能力共用。

## 错误处理

### 目标解析失败

如果没有找到联系人或群：

- 返回 `target_selection_card`
- 不直接执行
- 不静默 fallback

### 飞书权限不足

返回清晰错误：

- 缺失的权限
- 建议动作
- 是否需要重新绑定

### 工具执行失败

返回结构化失败结果：

- 哪一步失败
- 失败原因
- 是否已部分成功

### 知识证据不足

企业知识问题：

- 明确提示“当前知识库证据不足”
- 不允许模型冒充企业事实回答

通用问题：

- 允许 `general_llm` 回答
- 但在模式上标记为非企业知识结论

## 测试策略

### 后端

1. Intent Router
- 知识问答
- 纯动作
- 问答后动作
- 参数缺失澄清

2. Target Resolver
- 联系人 Top 3 推荐
- 群聊 Top 3 推荐
- `其他` 分支

3. Preview Card
- 预览生成
- 编辑后写回
- 确认执行按最新值执行

4. 动作执行
- 成功
- 失败
- 部分成功

### 前端

1. 目标选择卡渲染
2. 可编辑预览卡渲染
3. 编辑后再次确认
4. 取消执行

## 实施顺序

1. 先实现 `Intent Router`
2. 再实现 `Target Resolver`
3. 再实现可编辑预览卡
4. 最后把 HR skill 中的动作拼装逻辑迁移到通用 `Action Planner`

## 结论

这次设计不是继续“修补 HR 技能里的关键词判断”，而是把 `/ask` 提升成更通用的对话编排层：

- 用 `Intent Router` 理解用户在说什么
- 用 `Target Resolver` 找出真正目标
- 用可编辑预览卡让用户掌握将要发生的结果
- 再通过现有飞书工具执行

这样既满足当前的 HR 和飞书协同场景，也为后续按飞书官方插件思路继续扩展 tool / skill 留下清晰边界。
